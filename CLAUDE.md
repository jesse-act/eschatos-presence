## Stack

- **Vite + React 18 + TypeScript** (SPA, not Next.js)
- **React Router DOM v6** for client-side routing
- **Tailwind CSS** + **shadcn/ui** (Radix primitives) for styling and components
- **React Hook Form** + **Zod** for forms and validation
- **TanStack React Query v5** (installed, not yet used — ready for API integration)

## Architecture

### Routing

All routes live in `src/App.tsx`. The root `<Layout>` wraps all pages and renders `<Outlet>`. Pages live in `src/pages/`.

### i18n

Language switching uses a custom React Context:

- `src/i18n/LanguageContext.tsx` — provides `{ t, lang, setLang, toggleLang }`
- `src/i18n/translations.ts` — all strings in one nested object (`en` + `fr`)
- Persisted to `localStorage` (key: `"eschatos-lang"`)
- No automatic pluralization — all strings are hardcoded per locale
- **Always add both `en` and `fr` keys** when adding new text

Usage in any component:
```tsx
const { t } = useLanguage()
// t.nav.home, t.hero.title, etc.
```

### Design System

Custom Tailwind palette defined in `tailwind.config.ts`:
- `navy` — primary brand color
- `gold` — accent
- `cream` — soft background

CSS custom properties in `src/index.css` map these for shadcn compatibility. Prefer these semantic tokens over raw Tailwind colors.

### Component Conventions

- Shared layout: `src/components/Layout.tsx` (Navbar + Outlet + Footer)
- Reusable UI primitives: `src/components/ui/` (shadcn components — do not modify unless customizing)
- Page-level hero: `src/components/PageHero.tsx`
- `src/lib/utils.ts` exports `cn()` (clsx + tailwind-merge) — always use for conditional classes

### Forms

Contact form in `src/pages/Contact.tsx` is the reference pattern:
- `useForm` + `zodResolver` for validation
- Submission is currently a mock (setTimeout) — real API call goes in `onSubmit`

### Data Fetching

TanStack Query is configured in `src/main.tsx`. When adding API calls, use `useQuery`/`useMutation` from `@tanstack/react-query`. Static data currently lives inline inside pages.

### awesome-design-md collection

`D:\eschatos-presence\awesome-design-md\design-md\` contains 69 brand design systems (Stripe, Vercel, Linear, Figma, Airbnb, Notion, Supabase, etc.). Each folder has a `README.md` with a link to fetch the full `DESIGN.md` from `https://getdesign.md/<brand>/design-md`.

When the task requires UI that matches a specific aesthetic (e.g. "make this feel like Stripe"), fetch the relevant design file:
- Use the Firecrawl MCP (`firecrawl_scrape`) to retrieve it, or
- Reference `awesome-design-md/design-md/<brand>/README.md` for the URL

## Firecrawl MCP

Configured as an HTTP MCP server (remote hosted). Available tools:
- `firecrawl_scrape` — scrape a URL to clean markdown
- `firecrawl_search` — web search with full page content
- `firecrawl_crawl` / `firecrawl_check_crawl_status` — async site crawl
- `firecrawl_extract` — structured data extraction with schema
- `firecrawl_agent` / `firecrawl_agent_status` — autonomous web research
- `firecrawl_map` — discover all URLs on a site

Use Firecrawl to fetch DESIGN.md files, check library docs, or research UI patterns.

## graphify

Knowledge graph stored at `graphify-out/`. The graph updates **automatically**:
- After every file edit (PostToolUse hook runs `graphify update .` in background — AST-only, no API cost)
- After every git commit / checkout (post-commit + post-checkout git hooks)

Rules:
- Before answering architecture or codebase questions, check `graphify-out/GRAPH_REPORT.md` first
- If `graphify-out/wiki/index.md` exists, navigate it instead of reading raw files
- For cross-module questions, prefer `graphify query "<question>"`, `graphify path "<A>" "<B>"`, or `graphify explain "<concept>"` over grep
- To build the initial graph (first time only): run `/graphify .` or `graphify extract . && graphify build .`
